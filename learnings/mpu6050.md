# MPU6050 — MEMS IMU (Inertial Measurement Unit)

## What it is

The MPU6050 is a 6-axis sensor combining:
- **3-axis accelerometer** — measures linear acceleration (including gravity)
- **3-axis gyroscope** — measures angular velocity (rotation rate)

It communicates over I2C (SDA/SCL) and is one of the cheapest IMUs available. On Arduino Uno/Nano, that's pins A4 (SDA) and A5 (SCL).

## How the gyroscope works (MEMS)

Inside the chip is a tiny vibrating mass on a silicon structure (Micro-Electro-Mechanical System). When the chip rotates, the Coriolis effect deflects this vibrating mass sideways. Capacitive plates detect this deflection and output a voltage proportional to the **angular velocity** — how many degrees (or radians) per second the chip is spinning.

Key point: the gyro measures **rate of rotation**, not angle. To get an angle, you must integrate the rate over time — which is where drift becomes a problem.

## Raw output and sensitivity

The gyro outputs a 16-bit signed integer (-32768 to +32767) for each axis. The mapping from raw counts to real °/s depends on the configured **full-scale range**:

| GYRO_CONFIG register (0x1B) | Range    | Sensitivity (LSB per °/s) |
|------------------------------|----------|---------------------------|
| 0x00                         | ±250°/s  | 131                       |
| 0x08                         | ±500°/s  | 65.5                      |
| 0x10                         | ±1000°/s | 32.8                      |
| 0x18                         | ±2000°/s | 16.4                      |

**Tradeoff:** Narrower range = more precision (more LSBs per °/s) but saturates if you spin too fast. Wider range = handles fast rotation but coarser precision.

Our robot uses ±250°/s (0x00), which is good for a slow ground robot.

### Converting raw to real units

```
real_degrees_per_sec = raw_value / sensitivity
real_rad_per_sec     = raw_value / sensitivity * (pi / 180)
```

For ±250°/s:
```
GYRO_SCALE = (pi / 180) / 131 = 0.000133 rad/s per LSB
```

Example: raw gz = 655 → 655 × 0.000133 = 0.0873 rad/s = 5°/s

## Accelerometer sensitivity

Similarly for the accelerometer:

| ACCEL_CONFIG register (0x1C) | Range | Sensitivity (LSB per g) |
|-------------------------------|-------|-------------------------|
| 0x00                          | ±2g   | 16384                   |
| 0x08                          | ±4g   | 8192                    |
| 0x10                          | ±8g   | 4096                    |
| 0x18                          | ±16g  | 2048                    |

Our robot uses ±2g (0x00). Conversion: `real_m_per_s2 = raw / 16384 * 9.80665`

## Zero-rate offset (gyro bias)

This is a critical concept. Even when the MPU6050 is perfectly still, the gyro does NOT output exactly zero. It might read +3 or -7 on the raw scale. This is called **bias** or **zero-rate offset**.

Typical MPU6050 bias: around ±20 LSB, which at ±250°/s sensitivity is about ±0.15°/s.

This sounds tiny, but when you integrate it to get angle:
```
After 1 second:   0.15° of phantom rotation
After 10 seconds: 1.5°
After 1 minute:   9°
After 10 minutes: 90°
```

### How to fix it: startup calibration

At boot, while the robot is stationary:
1. Read the gyro N times (e.g., 200-500 samples)
2. Average each axis
3. Subtract that average from all future readings

This eliminates the static bias. The bias can also drift with temperature, but startup calibration handles the bulk of it.

## Digital Low-Pass Filter (DLPF)

Register 0x1A (CONFIG) controls an internal filter that smooths the raw output:

| DLPF_CFG | Accel bandwidth | Gyro bandwidth |
|----------|-----------------|----------------|
| 0        | 260 Hz          | 256 Hz         |
| 1        | 184 Hz          | 188 Hz         |
| 2        | 94 Hz           | 98 Hz          |
| 3        | 44 Hz           | 42 Hz          |
| 4        | 21 Hz           | 20 Hz          |
| 5        | 10 Hz           | 10 Hz          |
| 6        | 5 Hz            | 5 Hz           |

Our robot uses DLPF_CFG = 3 (44 Hz accel, 42 Hz gyro). This filters out high-frequency vibration from the motors while still allowing real motion signals through. Lower = smoother but adds latency.

## Axis orientation

The MPU6050 has a specific axis convention printed on the chip. For a robot:
- **Z-axis gyro** = yaw (turning left/right) — this is the one we care most about
- **X-axis gyro** = roll (tilting side to side)
- **Y-axis gyro** = pitch (tilting forward/back)

The physical mounting orientation of the chip on the robot matters. If you mount it rotated 90°, the axes swap.

## I2C communication

- Default address: 0x68 (AD0 pin low) or 0x69 (AD0 pin high)
- Supports standard (100 kHz) and fast (400 kHz) I2C — our code uses 400 kHz
- All sensor registers are in a contiguous block starting at 0x3B:
  - 0x3B-0x40: Accel X, Y, Z (6 bytes)
  - 0x41-0x42: Temperature (2 bytes)
  - 0x43-0x48: Gyro X, Y, Z (6 bytes)
- Reading all 14 bytes in one burst is most efficient and ensures the data is from the same sampling instant

## Why we don't use accelerometer for position

Accelerometer measures acceleration. To get velocity you integrate once; to get position you integrate twice (double integration). Each integration step amplifies noise and bias:

```
tiny accel bias → grows linearly in velocity → grows quadratically in position
```

Within seconds, the position estimate is meters off. This is why our EKF config does NOT fuse accelerometer data for position — it would be catastrophic. The accelerometer is only useful for estimating tilt (pitch/roll) via the gravity vector.
